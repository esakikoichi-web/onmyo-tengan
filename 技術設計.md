# 陰陽天眼 技術設計書

対象：完全放置サブスク（¥9,800/月・営業特化）。
前提の商品設計は `商品設計.md` を参照。

---

## 0. いま何が問題か（現状）

- `index.html` 1枚に**全ロジック＋全文章**が入っており、ソース表示で丸ごとコピー可能。
- `netlify.toml` の Basic認証（`admin:onmyo2026`）は覗き見防止のみ。課金の壁にならない。
- サーバ・DB・認証・決済がゼロ。データは端末内のみ。

→ 有料化には「①ロジックをサーバへ隠す ②会員と課金状態でゲート ③毎朝の自動配信」が必要。

---

## 1. 全体構成

```
[ブラウザ（薄いUI）]
   │  ①ログイン(LINE Login/LIFF)
   │  ②鑑定リクエスト(fetch)
   ▼
[Supabase]
   ├─ Auth（JWT。LINEログインを検証して発行）
   ├─ Postgres（客リスト・履歴・課金状態）
   ├─ Edge Functions（← ここに鑑定エンジンを隠す）
   │     • POST /reading    鑑定実行（要・課金チェック）
   │     • POST /customers   客リストCRUD
   │     • /webhook/stripe   課金状態を同期
   │     • /cron/morning     毎朝バッチ→LINE配信
   └─ pg_cron（毎朝6:00に /cron/morning を起動）
        │
        ├──→ [Stripe]  決済・トライアル・解約ポータル
        └──→ [LINE Messaging API]  毎朝の攻め順を配信
```

**技術選定の理由**：Supabase 1つで「認証＋DB＋サーバ関数＋定期実行」が揃う。
放置モデルに必須の「毎朝バッチ」を pg_cron で内包できるのが決め手。
フロントは現状の Netlify 静的ホストのまま流用。

---

## 2. コンテンツ保護（最重要）

**方針：価値ある計算と文章を Edge Function 側だけに置く。クライアントには結果しか渡さない。**

- `index.html` 内のエンジン関数（`erect` `topDays` `rToday` `rFull` `bestSlot`
  `banHtml` 等の算出・文章生成）を **`engine/` モジュールに抽出**し、
  **Edge Function からのみ import**。ブラウザには一切配布しない。
- クライアントは「入力を送り、返ってきたHTML断片を表示するだけ」の薄い殻にする。
- 暦データ・日干別トーク・悩み別文章（`SELF` `JOBWORRY` `SUIKJOB` 等の定数）も
  サーバ側へ移す。ここがコピーされたら商品価値が消えるため最優先。

### 移行を2段階に分ける（手戻り防止）
- **Stage 1**：認証＋課金ゲートを既存アプリの前に立てる（速い。ただしエンジンはまだ露出）。
- **Stage 2**：プレミアム部分のエンジンをサーバへ移設（本当の保護）。
- 無料の味見（自分1件）だけはクライアント計算のまま残してよい＝露出しても損失が小さい範囲。

## 3. データモデル（Postgres）

```sql
-- プロフィール（Supabase auth.users と1:1）
profiles(
  id            uuid pk references auth.users,
  line_user_id  text unique,        -- 毎朝配信の宛先
  display_name  text,
  job           text default '営業',
  self_birth    date,               -- 「自分を観る」用
  created_at    timestamptz default now()
)

-- 客リスト（放置モデルの継続エンジンの中核）
customers(
  id          uuid pk,
  user_id     uuid references profiles,
  name        text,
  birth_date  date not null,
  birth_time  time,                 -- 任意
  sex         text check (sex in ('男','女')),
  status      text,                 -- 商談段階メモ
  created_at  timestamptz default now()
)

-- 鑑定履歴（端末越しに残す＝乗り換えコスト）
readings(
  id          uuid pk,
  user_id     uuid references profiles,
  customer_id uuid references customers,
  type        text,                 -- attack/name/compat/kichi/shoubu...
  input       jsonb,
  result_html text,                 -- キャッシュ
  created_at  timestamptz default now()
)

-- 課金状態（Stripe のミラー）
subscriptions(
  user_id             uuid pk references profiles,
  stripe_customer_id  text,
  stripe_sub_id       text,
  status              text,         -- trialing/active/past_due/canceled
  current_period_end  timestamptz,
  trial_end           timestamptz
)

-- 配信ログ（二重送信防止・分析）
morning_log(
  user_id  uuid references profiles,
  date     date,
  sent_at  timestamptz,
  primary key (user_id, date)
)
```

**RLS（行レベルセキュリティ）**：`customers` `readings` は
`user_id = auth.uid()` の行だけ本人が読める設定を必ず入れる。

---

## 4. 課金ゲート（Stripe）

- **Stripe Checkout（ホスト型）** で加入。`trial_period_days: 7` で7日トライアル。
- 解約は **Stripe Customer Portal**（ホスト型）に丸投げ → 改正特商法の解約導線を自前実装せず満たしやすい。
- **Webhook** で `customer.subscription.*` を受け、`subscriptions` テーブルを更新。
- **ゲート判定**：`/reading` は毎回 `subscriptions.status in ('trialing','active')` を確認。
  無料の味見（自分1件）だけ未加入でも許可。

---

## 5. 会員・ログイン（LINE）

- **LINE Login（LIFF）** を採用。理由：毎朝配信に LINE userId が必須なので、
  ログインと配信先取得を同時に済ませられる。
- フロー：LIFF でログイン → IDトークンを Edge Function で検証 →
  Supabase の JWT を発行 → `profiles.line_user_id` を保存。
- **LINE公式アカウント（Messaging API）** の友だち追加を加入フローに必須化
  （追加しないと毎朝配信が届かない＝主力機能が動かないため）。

---

## 6. 毎朝の自動配信（放置モデルの心臓）

- **pg_cron** で毎朝 6:00 に Edge Function `/cron/morning` を起動。
- 処理：
  1. `subscriptions.status in ('trialing','active')` の全ユーザーを取得。
  2. 各ユーザーの `customers` を当日の暦で再計算 → 「今日攻める順」＋各人の一言＋本日の吉日を生成。
  3. LINE Messaging API（push message）で配信。
  4. `morning_log` に記録して二重送信を防ぐ。
- 客リストが空のユーザーには「まず客を登録して」の一言だけ送る（オンボーディング）。
- 失敗時リトライと、1日1通の冪等性（`morning_log` の主キーで担保）。

---

## 7. API 一覧（Edge Functions）

| メソッド/パス | 認証 | 課金 | 役割 |
|---|---|---|---|
| POST /reading | 要 | 要（味見除く） | 鑑定実行。エンジンはここだけ |
| GET/POST/DELETE /customers | 要 | 要 | 客リストCRUD |
| GET /readings | 要 | 要 | 履歴取得 |
| POST /auth/line | 不要 | - | LINEログイン検証→JWT発行 |
| POST /webhook/stripe | 署名 | - | 課金状態同期 |
| POST /cron/morning | 内部 | - | 毎朝バッチ |

---

## 8. 移行ロードマップ

1. **エンジン抽出**：`index.html` の算出/文章生成を `engine/` へ切り出し（挙動を変えずに分離）。
2. **Supabase 構築**：プロジェクト作成・上記スキーマ・RLS。
3. **薄いクライアント化**：UIは流用し、計算を `/reading` 呼び出しに置換。
4. **Stripe**：Checkout＋Portal＋Webhook。7日トライアル。
5. **LINE**：Login（LIFF）＋Messaging＋毎朝 pg_cron。
6. **法務ページ**：特商法表記・利用規約・プライバシーポリシー・占い免責。
7. **Basic認証を撤去**して一般公開。
